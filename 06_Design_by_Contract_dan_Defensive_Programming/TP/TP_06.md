## 1. Analisis Perbandingan: Asersi vs Eksepsi

Dalam pengembangan perangkat lunak modern, memastikan keandalan kode adalah prioritas utama. Dua mekanisme yang sering digunakan untuk menjaga integritas program adalah asersi dan eksepsi. Meskipun keduanya tampak serupa karena sama-sama berfungsi untuk mendeteksi kesalahan, tujuan penggunaan dan waktu implementasinya sangat berbeda. Pemahaman tentang kapan harus menggunakan salah satunya merupakan keterampilan krusial bagi seorang pengembang, terutama dalam menerapkan prinsip _Design by Contract_ (DbC).

**Asersi** adalah alat yang digunakan selama fase pengembangan dan debugging untuk memvalidasi asumsi internal pengembang. Asersi berfungsi untuk memeriksa kondisi yang dianggap "mustahil" terjadi jika kode ditulis dengan benar. Contohnya, jika sebuah fungsi menghitung luas lingkaran, asersi dapat digunakan untuk memastikan bahwa radius tidak pernah negatif. Jika asersi gagal, itu berarti ada _bug_ pada logika internal program. Di banyak bahasa pemrograman, asersi dapat dinonaktifkan pada fase produksi demi efisiensi performa, karena asersi seharusnya tidak pernah terpicu dalam perangkat lunak yang sudah matang dan bebas _bug_.

Di sisi lain, **Eksepsi** digunakan untuk menangani kondisi kesalahan yang "mungkin" terjadi saat program berjalan di tangan pengguna (_runtime_). Kesalahan ini biasanya berasal dari faktor luar yang tidak bisa dikontrol sepenuhnya oleh pengembang, seperti kegagalan koneksi internet, file yang hilang, atau input pengguna yang tidak valid. Berbeda dengan asersi yang menghentikan program secara paksa, eksepsi dirancang untuk ditangkap (`catch`) agar program dapat memberikan respon yang lebih elegan, seperti menampilkan pesan kesalahan yang ramah pengguna, alih-alih mengalami _crash_.

Pertanyaannya, apakah kita harus menggunakan sepenuhnya asersi atau sepenuhnya eksepsi? Jawabannya adalah tidak. Keduanya harus digunakan secara berdampingan namun dengan porsi yang tepat. Kita tidak bisa menggunakan asersi untuk memvalidasi input pengguna karena pengguna bisa saja melakukan kesalahan yang tidak terduga. Sebaliknya, menggunakan eksepsi untuk mendeteksi kesalahan logika internal (seperti variabel yang tiba-tiba bernilai _null_ karena kesalahan algoritma) hanya akan menyembunyikan masalah yang seharusnya diperbaiki oleh pengembang di level kode.

Sebagai kesimpulan, asersi adalah "penjaga gerbang" bagi pengembang untuk memastikan kontrak internal program terpenuhi, sedangkan eksepsi adalah "jaring pengaman" untuk menangani dinamika dunia nyata saat aplikasi dijalankan. Penggunaan yang seimbang antara keduanya akan menghasilkan perangkat lunak yang tangguh (_robust_), mudah dipelihara, dan memiliki pengalaman pengguna yang baik.

## 2. Implementasi Kode

Berikut adalah perbandingan implementasi fungsi pembagian menggunakan kedua metode tersebut:

### A. Menggunakan Asersi (Assertion)

Digunakan untuk memastikan input internal benar selama masa pengembangan.

```javascript
const assert = require('assert');

function divide(a, b) {
    assert(b !== 0, 'Asumsi Gagal: Tidak bisa pembagian dengan nol');
    return a / b;
}

// Output jika b = 0:
// AssertionError [ERR_ASSERTION]: Asumsi Gagal: Tidak bisa pembagian dengan nol
```

### B. Kode dengan Eksepsi (Exception)

```javascript
function divide(a, b) {
    if (b === 0) {
        throw new Error("Maaf, angka pembagi tidak boleh nol.");
    }
    return a / b;
}

try {
    console.log(divide(10, 0));
} catch (error) {
    console.error("Output:", error.message);
}

// Output jika b = 0:
// Output: Maaf, angka pembagi tidak boleh nol.
```
